# CopyQ feature requests глазами clipmgr

Разбор самых востребованных feature requests репозитория
[hluk/CopyQ](https://github.com/hluk/CopyQ) (метка `feature`, сортировка по
реакциям и по числу комментариев, дедуп). Цель — не скопировать CopyQ, а понять,
**что из народного вишлиста реально стоит взять** в наш минималистичный
`Win+V`-аналог, что осознанно оставить за бортом, и какие дешёвые победы мы
упустили.

Напоминание о позиционировании clipmgr: нативная история буфера для **GNOME/X11**,
резидентный GTK-демон, попап у курсора, `Enter` вставляет через XTEST. Сейчас —
**только текст, только в памяти** (диск не пишем намеренно). **Картинки — в
ближайших планах.** Всё, что тянет CopyQ в сторону IDE буфера обмена (скрипты,
табы, синхронизация, темы), для нас — bloat.

Вердикты: **Взять** — в scope, делаем; **Возможно** — приятно, но потом;
**Вне scope** — раздувание для `Win+V`-клона; **Уже есть** — реализовано.

## Дешёвые победы, которые мы упустили

Небольшие фичи с высоким соотношением польза/стоимость, ложащиеся прямо на нашу
архитектуру (попап + in-memory список + XTEST-вставка). Все не требуют диска и не
ломают минимализм.

1. **Поиск-по-набору (search-as-you-type)** — [#145](https://github.com/hluk/CopyQ/issues/145),
   [#1288](https://github.com/hluk/CopyQ/issues/1288), [#1834](https://github.com/hluk/CopyQ/issues/1834).
   Самое востребованное и самое дешёвое для нас. Клавиатуру мы и так читаем
   поллингом через xgb (не GTK), так что печатные символы уже приходят нам в
   руки — добавить строку-фильтр и перестраивать `listBox` по substring. Стрелки
   продолжают двигать выбор (фокус «остаётся» в поиске — у нас его физически и нет,
   мы рулим выделением вручную через `SelectRow`). **Наивысший приоритет.**

2. **Цифровые быстрые хоткеи `1`–`9`** — [#246](https://github.com/hluk/CopyQ/issues/246).
   В открытом попапе цифра мгновенно вставляет N-ю видимую запись. Одна ветка в
   обработчике клавиш поллинга → `finish(true)` с нужным индексом. С поиском
   комбинируется (цифры вставляют, если не в режиме ввода запроса, либо по
   модификатору). Очень «Win+V».

3. **`max-entries` cap + дедуп** — [#2244](https://github.com/hluk/CopyQ/issues/2244),
   [#344](https://github.com/hluk/CopyQ/issues/344).
   Раз память конечна, кольцевой буфер на N записей (например, 200) + отбрасывание
   точного дубликата (поднимать существующую запись наверх вместо добавления).
   Тривиально на срезе в памяти; сразу делает историю чистой.

4. **Pin / favorites + «очистить всё, кроме закреплённого»** —
   [#1172](https://github.com/hluk/CopyQ/issues/1172), [#2964](https://github.com/hluk/CopyQ/issues/2964).
   Флаг `pinned` на записи: не вытесняется по `max-entries`, показывается сверху.
   Хоткей закрепления в попапе. Практически «даром» на in-memory модели.

5. **Clear-history / clear-all**. В CopyQ подразумевается везде; у нас нет вовсе.
   Хоткей (или пункт) «очистить историю» — одна строка, обнуляющая срез (с учётом
   pinned). Плюс опционально авто-очистка при рестарте
   ([#2918](https://github.com/hluk/CopyQ/issues/2918)) — у нас она уже де-факто есть
   (память не переживает перезапуск), можно просто задокументировать как фичу.

6. **Исключение секретов / источников-паролей** —
   [#3014](https://github.com/hluk/CopyQ/issues/3014), FAQ CopyQ про «ignore password window».
   Раз мы владеем буфером и знаем момент копирования, можно не сохранять записи,
   когда активное окно — известный менеджер паролей (по WM_CLASS), либо когда в
   `TARGETS` буфера присутствует маркер секрета (`x-kde-passwordManagerHint`,
   которым помечают пароли KeePassXC/ряд GTK-приложений). Дёшево, ощутимо для
   доверия. CLI-инструменты (`pass`) сложнее — но хотя бы app-level exclusion берём.

7. **«Вставить как plain text»** — [#151](https://github.com/hluk/CopyQ/issues/151).
   Мы уже text-only, так что дефолт и так plain. Осмысленный жест — alt-хоткей
   вставки, который принудительно кладёт `text/plain` без наследуемого богатого
   формата (актуально станет с картинками/rich-источниками). Пока — почти бесплатно.

8. **Undo последнего удаления** — [#187](https://github.com/hluk/CopyQ/issues/187).
   Держать last-deleted запись и возвращать по хоткею. На in-memory списке —
   тривиально; заменяет диалог подтверждения ([#2253](https://github.com/hluk/CopyQ/issues/2253)).

9. **Вставка через «набор текста» (autotype)** — [#3444](https://github.com/hluk/CopyQ/issues/3444).
   Для окон, где Ctrl+V не работает (VNC/Citrix/VM/некоторые терминалы), режим
   «печатать содержимое посимвольно» через XTEST. У нас уже есть раскладко-независимый
   инжект (запасной keycode) — это надстройка над `pasteInto`, отдельным хоткеем.

## Image-related requests (мы добавляем картинки)

Картинки — ближайший план, поэтому эти тикеты стоит держать перед глазами как
чек-лист ожиданий пользователей:

- [#343](https://github.com/hluk/CopyQ/issues/343) **Картинки как записи** (14 комм.) —
  базовый кейс: сохранить image из буфера как запись и уметь её вставить обратно.
  Ядро фичи.
- [#252](https://github.com/hluk/CopyQ/issues/252) **Images: сохранить в файл** (45 комм., самый
  обсуждаемый по картинкам) — из попапа «сохранить как PNG». Дёшево и ожидаемо.
- [#2456](https://github.com/hluk/CopyQ/issues/2456) **Несколько картинок** (12 комм.) — история
  должна нормально держать серию изображений, не только последнее.
- [#1687](https://github.com/hluk/CopyQ/issues/1687) **Удалить все картинки, оставить текст**
  (15 комм.) — раз появляются разные типы, нужен фильтр/массовое удаление по типу.

Практические выводы для нашей реализации: (1) записи станут типизированными
(text/image) — превью-миниатюра в строке попапа вместо 3 строк текста; (2) картинки
тяжелее — `max-entries` (дешёвая победа №3) становится обязательным, иначе память
поедается; (3) владение буфером усложняется: отдавать `image/png` в ответ на запрос
вставки наравне с `text/plain`.
